# Hogwarts Loop

> 一个「规划 → 构建 → 评审 → 提交」的循环纪律，把一项大型工程从计划一路推进到**已提交、已评审**的代码——方式是把它拆成一连串**小而可独立验证**的检查点（checkpoint）。

这是一个 Claude Code **Skill**（技能）。它不是一段能直接运行的程序，而是一套写给编排型 Agent 的**工作方法与角色协议**：当你要执行一项「错了代价很大、爆炸半径很高」的工程时，用它来组织多个 Agent 分工协作、互相制衡。

## 一句话概括

把一个庞大、模糊的工程，拆成小而可独立验证的**检查点**；每个检查点由**实现者 Agent（Harry）**编写代码，再由一个**独立的评审员 Agent（Severus，以及一组专家）**尝试把它攻破，两者循环「修复 → 重审」直到干净；然后由**编排者（Albus，也就是你/主 Agent）**提交它，再推进到下一个检查点。

核心引擎是 **构建者 / 评审员分离**：写代码的人和挑毛病的人，永远不是同一个 Agent。

---

## 为什么值得这份开销

跑两个 Agent 外加一个循环，比你自己直接实现要贵。它恰恰在**「出错代价高」**时才回本，因为它防的是真实工程里最大的失败模式：

> **一个作者，看不见自己刚刚说服自己「是对的」的代码里的 bug。**

实现者自己的测试之所以通过，是因为它们只测了实现者**想到过**的东西。而一个抱着对抗心态、重新从规范推导正确性的独立评审员，会去走那些作者跳过的路径。**最糟糕的 bug 就住在那道缝里——那些「测试全绿却带病上线」的 bug。**

所以这套方法的价值不是「多评审一遍」，而是三件事的组合，缺一不可，少一件就退化成走过场：

1. **结构性独立**——评审员不是作者。
2. **有界的爆炸半径**——检查点足够小。
3. **每次改动都重跑一遍的全量安全网**——门禁（gates）。

## 三个角色

这套阵容是一个助记法：邓布利多（Albus）在上面运筹、部署众人；哈利（Harry）干活；斯内普（Severus）审视一切。**派发任务时就用这些名字**，好让各个 Agent 保持独立、各自积累自己的上下文。

| 角色 | 名字 | 是谁 | 职责 | 绝不做的事 |
|------|------|------|------|-----------|
| **规划者 / 编排者** | **Albus** | **就是你**（主 Agent），不是被派生的 Agent | 读「真相源」→ 拆成检查点计划 → 派发、裁决、提交、把备忘录（memo）和偏差（deviation）往下传 | **绝不亲手写功能代码**——一旦规划者也去实现，就没有独立裁判了 |
| **实现者** | **Harry** | 一个贯穿全程的持久 Agent | 按规范实现**一个**检查点、跑门禁、如实汇报所做的一切与每一处偏差、留在自己的范围内 | 不提交、不越界 |
| **评审员** | **Severus**（+ 专家席：Alastor / Filius / Minerva） | 一个贯穿全程的持久 Agent | 对照规范独立验证、**自己重跑门禁**、对抗式地试图攻破、给出 APPROVE 或 REQUEST-CHANGES 并分级列出问题 | **只裁决，从不修**（Harry 才修） |

> **把 Harry 和 Severus 保持为两个持久、带上下文的独立 Agent**，让他们跨检查点积累项目记忆（Severus 记得自己标记过什么；Harry 记得自己的决策）。每个检查点**复用同样这两个 Agent**，而不是每次都新开。只有在用户明确要求这套流程时才派生它们——它是**可选加入（opt-in）**的。

## 什么时候用，什么时候别用

### ✅ 适用

- 对照设计文档 / 规范，**规划并构建**里程碑 / 阶段性工作
- 大型重构、引擎 / 框架内部实现
- 任何横跨多个文件、多个会话的工作
- 任何「一次静默的回归代价很高」的场景

> 在**规划阶段**就该拿出它——当用户想铺开一大块工作**然后真的把它建出来**时，而不是等代码已经在写了才想起来。

### ❌ 不适用

- 一次就能做完并验证的小改动
- 目标是「学习」而非「交付」的探索性尝试（spike）
- **没有客观门禁**的任务（没有办法说「这个检查点通过了」，你只会空转）
- 对一个已有 PR 做一次性评审
- 撰写设计 / 需求规范本身（那是这个循环的**输入**，不是它的工作）

## 工作机制：核心循环

整个机制是一个状态机：先有一次**规划**，然后对每个检查点跑一个**「构建 → 评审 → 修复」**的小循环，全部通过后**收尾**。

```
                          ┌──────────── 规划 — Albus（一次）────────────┐
                          │ 读真相源 → 拆成检查点 → 写计划文档 →          │
                          │ 为每个检查点定义门禁                          │
                          └──────────────────────┬──────────────────────┘
                                                 │
        ┌────────────── 每个检查点（按依赖顺序）───────────────┐
        │                                                       │
        │   ALBUS（编排者）─[A: 实现简报]─▶ HARRY（实现者）      │
        │        ▲                          │                   │
        │        │                    构建 + 跑门禁              │
        │        └──────[B: 交付报告]───────┘                   │
        │        │                                              │
        │   编排者 ──[C: 评审简报]──▶ SEVERUS（评审员）          │
        │        ▲                       │                      │
        │        │               验证 + 重跑门禁 + 探测          │
        │        └────────[D: 评审报告]──┘                      │
        │        │                                              │
        │   裁决？                                              │
        │     ├── REQUEST-CHANGES ─[E: 问题转达]─▶ HARRY        │
        │     │        ▲                            │           │
        │     │        └──────[F: 修复报告]─────────┘           │
        │     │        └──────▶ 回到 [C]（Severus 重审）         │
        │     │                                                 │
        │     └── APPROVE ──▶ 编排者提交（代码 + 计划更新）+     │
        │                     把 P3/备忘录往下传 + 推进到下一个   │
        └───────────────────────────────────────────────────────┘
                                                 │
                          ┌──────────────────────┴──────────────────────┐
                          │ 收尾：验收映射 · 把偏差并回规范 ·             │
                          │ 收尾文档 · 明确点名残余 / 仅手工的风险         │
                          └──────────────────────────────────────────────┘
```

**一个检查点只有在两个条件同时满足时才算通过：Severus 批准（APPROVE）** 且 **门禁全绿**。两者，每次都要。

### 三个阶段拆解

**1. 规划（Albus 做一次，也是全局最高杠杆的决策）**

- **读真相源。** 设计 / 规范 / 路线图是下游一切都要引用的契约。如果它不存在，这是个危险信号——本循环需要一个客观标准来对照评审。
- **拆成检查点。** 按「可独立验证的能力」切分（不是按文件、按层切）；把基础 / 基建检查点放最前；在有风险的重写之前先铺一个回归基线；把破坏性 / 高风险的工作集中进它自己的检查点；按依赖排序；以一个收尾检查点结束。**典型的里程碑落在 5～9 个检查点**。
- **为每个检查点预先定义门禁**——让「通过」成为客观的命令 + 断言，而不是一种感觉。
- **写下活的计划文档**（见 `assets/plan-template.md`），并**先把它提交**——它是整个运行的脊梁，也是各阶段之间的记忆。

**2. 每个检查点的小循环**

1. 给 Harry 派发**实现简报**（边 A）。
2. 收到 Harry 的**交付报告**（边 B）。
3. 给 Severus / 某位专家派发**评审简报**（边 C）。
4. 收到**评审报告**（边 D）。
5. 若 **REQUEST-CHANGES** → 转达阻塞性问题（边 E）→ Harry 修复（边 F）→ 重审（回到边 C）。循环直到干净。**绝不带着未解决的 P1/P2 推进。**
6. 若 **APPROVE 且门禁全绿** → 把这个检查点作为**一个可追溯的单元**提交（代码 **和** 计划文档的状态更新在同一个提交里）→ 给该检查点盖上 ✅ + 日期 + 提交哈希 → 把备忘录往下传 → 推进。

**3. 收尾**（见下文[收尾](#收尾)）。

## 数据交接：每条边上传递什么

**交接处是松散运行的循环会漏东西的地方。** 上图每一条带字母的边都携带一份特定的载荷（payload）。把它们当作角色之间的契约来对待——**如果一份载荷没被明确写下，它就容易被丢掉。**

| 边 | 方向 | 载荷 | 关键内容 |
|----|------|------|---------|
| **A** | 编排者 → Harry：**实现简报** | 本检查点的范围、确切的规范章节引用、门禁命令、硬约束、**从早期检查点带来的备忘录** | 点名确切的规范章节，比复述需求更重要——它让实现者和评审员引用同一个标准 |
| **B** | Harry → 编排者：**交付报告** | 交付了什么（对照简报清单）、**完整的门禁真实输出（包括难看的部分）**、每一处偏差（ID + 改了什么 + 为什么）、改动的文件、自己标记的高风险点、「未提交 / 未越界」的确认 | 「自我标记要重点审的地方」反而让你显得更好——评审员恰恰在作者已隐约不安的地方，更容易找到真 bug |
| **C** | 编排者 → Severus：**评审简报** | 怎么看 diff + 基线提交、要对照的规范章节、Harry 声明的偏差清单、按优先级排的关注区 | 只审未通过的改动 |
| **D** | Severus → 编排者：**评审报告** | 裁决（APPROVE / REQUEST-CHANGES）、每条问题（[P1/P2/P3] + 文件:行 + 具体失败场景 + 建议修法）、对每处偏差的裁定、门禁重跑结果（诚实说明哪些只是静态分析）、APPROVE 时给后续检查点的**下游备忘录** | — |
| **E** | 编排者 → Harry：**问题转达** | 要修的 P1/P2 问题（P3 只记录、不阻塞）、哪些偏差被驳回必须改、编排者对争议点的裁决 | — |
| **F** | Harry → 编排者：**修复报告** | 每条问题的修法、**重跑整套门禁**（不是只跑动过的模块）、为锁住每个 bug 而新增的测试 | 然后回到边 C 让 Severus 重审 |

### 严重度分级（P1 / P2 / P3）

分级是让循环既不在鸡毛蒜皮上卡壳、也不放真 bug 过关的关键：

- **P1 — 阻塞，正确性。** 一个真实缺陷。要给出**具体失败场景**（确切的输入 / 状态 → 错误输出或崩溃），不是泛泛的担忧。必须修复才能通过。
- **P2 — 阻塞，应修。** 规范违背、缺失的必需测试、契约不符、验证不了任何东西的重言式断言。除非明确豁免，否则修复才能通过。
- **P3 — 非阻塞。** 风格、文档措辞、面向未来的改进、留给后续检查点的观察。记录下来带走，不为它阻塞。

**当 P1 和 P2 全部清零、且门禁全绿时，检查点通过。** P3 搭便车留在计划文档的「往下传的备忘录」里。

## 跨检查点的记忆：计划文档

有两样东西必须跨越检查点边界持续存在，否则循环会漏掉知识。二者都写在**计划文档**里：

- **偏差登记表（deviation registry）。** 每一处被接受的偏差，带 ID、原因、范围。这是「我们实际建的 vs. 文档所说的」的诚实记录，也是收尾时你要并回规范 / 契约的东西。
- **往下传的备忘录（carry-forward memos）。** 当 Severus 的 APPROVE 报告（边 D）里带着一条下游备忘录——下一阶段必须遵守的约束、要绕开的限制、终章必须补的测试——编排者**此刻就**把它写进那个未来检查点的「待生成简报」里。一个读过整个运行的评审员，常常能提前好几个检查点预测到一处隐患；把预测捕捉下来，否则你会用最难的方式重新发现它。

> **如果它没被写进计划文档，它就等于没发生。** 这份活文档就是循环在各阶段之间的记忆。

## 评审员的变体：教授席位

评审员是一个**角色**，不是一个固定身份。唯一的不变量是**独立性：评审员绝不能是作者。** 只要这条守住，你就可以改变「谁来审、几个人审、用什么工具审」。**异质的评审员是优点**——不同评审员有不同盲区，一组人拓宽的是覆盖面而非重复。

### 专家席位

Severus 是默认的主审（正确性 / 对抗）。对于风险落在特定视角上的检查点，加入（或换上）一位专家。每人以一位霍格沃茨教授命名，其学科 / 性格映射到该视角，名字本身就是助记（Albus / 邓布利多坐在席位**之上**作为编排者，不在席位内）：

| 评审员 | 视角 | 盯什么 |
|--------|------|--------|
| **Severus**（斯内普，魔药） | 正确性 / 对抗（默认主审） | 测试没覆盖的路径；作者看不见的 bug——无一能逃过他 |
| **Alastor**（疯眼汉穆迪，傲罗） | 安全 | 注入、授权、密钥、不安全反序列化、不可信输入、攻击面——「时刻警惕」 |
| **Filius**（弗立维，魔咒；决斗冠军） | 性能 | 热路径、分配、复杂度、N+1、无谓的工作——又快又优雅 |
| **Minerva**（麦格，变形术） | 并发 | 竞态、死锁、重入、状态转移顺序——秩序与纪律 |

在风险与视角匹配的检查点上价值最高（例如 Alastor 审触碰鉴权或解析不可信输入的检查点；Minerva 审加入线程或调度器的；Filius 审热路径的）。席位是可扩展的——需要时为另一个视角再加一位教授。

### 其他形态

- **评审小组（多位评审员，同一个检查点）。** 对同一个 diff 派发 2～3 位教授；各自返回自己的报告（边 D），编排者去重合并后再转达给 Harry。**保留给最高风险的检查点**。
- **把内置评审工具当作评审员或额外门禁。** 若宿主提供评审命令 / 技能——如工作区 diff 的 bug 评审（`/code-review`，可调更高强度档）、安全扫描（`/security-review`）、图谱支撑的结构评审——可把其中一个作为检查点门禁的一部分来跑，或作为 Severus 旁边的独立第二意见。
- **重量级云端评审（ultrareview）。** 多 Agent 深度评审（`/code-review ultra`）是「成败在此一举」检查点的最强门禁。**注意：它由用户触发且计费——编排者无法自行发起。** 所以由人在想额外加固的检查点上运行它，编排者再像对待任何评审报告一样把它的发现纳入循环。

> 无论怎么组合，都要保住那条不变量与纪律：**独立于作者、分级列出问题、自己重跑门禁、如实说明验证边界。**

## 收尾

（在最后一个检查点之后）

- **编制验收映射：** 每一条需求 ↔ 它是如何被验证的（自动化测试 / 无头断言 / 对环境无法执行的东西标为**手工或用户验收**——如实说明，绝不伪造）。
- **把偏差登记表并回规范 / 契约**作为勘误。
- **写收尾文档：** 验收勾选、已知限制（记录，不隐藏）、明确推迟到未来里程碑的条目。
- 自动化安全网抓不到的残余风险，靠**明确点名**来处理，而不是假装它被覆盖了。

---

## 反模式：循环是怎么烂掉的

- **编排者写代码。** 就没有独立裁判了——你在替身式地评审自己的工作。
- **同一个 Agent 又实现又评审。** 直接毁掉这套方法唯一赖以成立的属性。
- **Severus 轻信 Harry 的门禁报告。** 永远自己重跑一遍。
- **带着未解决的 P1/P2 推进。** 整件事的意义就是不把已知的 bug 发出去。
- **伪造你没跑成的验证。** 说清边界，路由到手工验收。
- **在检查点之间弄丢一条备忘录。** 不在计划文档里，就等于没了。
- **检查点大到无法深入评审。** 如果 Severus 没法把它整个装进脑子里，就切得更细。

## 如何适配到你的项目

本方法与语言、框架无关。把下面这些「旋钮」绑定到手头的项目：

- **门禁命令**——在这里「绿」意味着什么（例如 `cargo test --workspace` + clippy + fmt；或 `pytest` + `ruff` + `mypy`；或一次构建 + e2e 套件）。
- **真相源**——评审所引用的设计文档、RFC、路线图或工单。
- **沟通语言**——与用户保持一致（例如项目惯例是中文就用中文回复），同时代码 / 注释遵循仓库约定。
- **仓库工具**——如果项目有代码智能层（知识图谱、评审 MCP 等），告诉 Severus 优先用它而不是盲扫文件；更快，也能给出结构性上下文。
- **「文档先行」的项目**——如果惯例是「先设计文档再写代码」，计划文档就作为设计与收尾之间的执行追踪层嵌进去。

## 一句话记住它

> 绿色的测试意味着「作者想到要测的东西通过了」，不意味着「代码是对的」。真正带病上线的 bug，是作者没想到的路径上的那些——空闲路径、空集、重入调用、并发窗口。一个专门去**找**这些路径的独立评审员，就是抓住它们的机制。这个技能里其余的一切，都是为了支撑这一个动作。
